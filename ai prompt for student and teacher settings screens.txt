Part 1: Core Architectural Principles (Must Follow)
Presentation Screen, Not Fetching Screen: The screen should receive pre-fetched user data via a constructor or a provider (e.g., from a Cubit/Bloc/Provider). It should not contain API call logic to fetch its own data. Assume data is provided by a parent widget or a state management container.

Global State for App Settings: Theme (Dark/Light) and Language are Application-wide settings. The screen must interact with a Global State Management system (e.g., ThemeCubit, LocaleCubit) to read and update these values. Do not manage theme/locale locally within the screen.

Local-First, Server-Sync (for Preferences): When a user toggles Dark Mode or changes Language:

The change must be immediate in the UI.

The preference must be saved locally (e.g., SharedPreferences).

Background Sync: After saving locally, the app should attempt to sync this preference to the backend user profile in the background. The screen should not wait for this sync to complete.

Centralized Localization: All text strings must be pulled from a centralized localization system (e.g., AppLocalizations.of(context)!.settings_title). Hard-coded strings are strictly forbidden.

Theming: All colors and styles must be derived from the main Theme.of(context) and colorScheme. Hard-coded colors are strictly forbidden. The UI must respond correctly to both Light and Dark themes.

Modular Navigation: The screen is a Hub. Tapping on any tile (e.g., "Change Password", "Edit Profile") should navigate to a dedicated, separate screen. Do not implement the detailed logic (e.g., the password change form) inside this screen.
