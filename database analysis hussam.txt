EduLearn Platform: Database Architecture Reference
Document Version: 1.0
Prepared for: Technical Team & Stakeholders

1. Executive Overview
The EduLearn platform employs a Dual-Database Architecture designed for separation of concerns:

Management Database (edulearn_db) : Handles master data, authentication for dashboard users, and academic structure skeleton.

Application Database (edulearn_app) : Handles operational educational content, AI authoring, lesson delivery, assessments, and real-time communication.

4. Summary of Relationship Types
Relationship Type	Implementation Strategy	Example
One-to-Many (1:N)	Standard Foreign Key constraint (within same DB)	lessons.id → lesson_blocks.lesson_id
Cross-Database Logical FK	Application-layer enforced references (IDs only)	app_db.lessons.teacher_id → mgmt_db.teachers.id
Many-to-Many (N:N)	Pivot Table with composite unique keys	teacher_class_subjects
Polymorphic	sender_type + sender_id columns	messages (Sender can be Teacher OR Student)
Versioned / Temporal	Self-referential link to previous version	lesson_exercise_versions.previous_version_id
5. Architectural Considerations & Recommendations
Cross-Database Integrity: Currently, there are no foreign key constraints between edulearn_db and edulearn_app. This provides flexibility but requires strict service-layer validation to prevent orphaned records (e.g., a lesson referencing a deleted class_section).

Authentication Model: The system uses two separate authentication sources (edulearn_db.users for admins, edulearn_db.students / edulearn_db.teachers for app users). Consolidation into a single Identity Provider is recommended for future security and Single Sign-On (SSO) capabilities.

Messaging Redundancy: The presence of both conversations/messages and human_chats tables indicates an evolving or duplicated communication layer. A refactor to unify this into a single, robust chat service is advised.

Assessment Scalability: The Versioned Assessment Model (drafts -> versions -> version_items) is robust and suitable for high-stakes tracking and analytics. It ensures that student answers are always tied to a specific, immutable snapshot of the questions.
