# تصميم فني لميزة إدارة الفروع (Branch Management)

## 1. تحليل المتطلبات

تتطلب الميزة الجديدة السماح لمشرف المدرسة (School Admin) بإضافة فروع جديدة لمدرسته. يجب أن يتضمن كل فرع جديد بيانات كاملة مشابهة لبيانات المدرسة الرئيسية، مع إنشاء حساب مشرف فرع (Branch Admin) بكلمة مرور مؤقتة تتطلب التغيير عند أول تسجيل دخول. يجب أن تخضع جميع طلبات إضافة الفروع لموافقة المشرف العام (Super Admin)، مع احتفاظ مشرف المدرسة الرئيسية بالتحكم الكامل والصلاحيات على حسابات الفروع.

## 2. تحديث مخطط قاعدة البيانات (Database Schema Updates)

لتحقيق هذه المتطلبات ضمن هيكلية Laravel الحالية، سنقوم بتعديل جدول `Schools` وإضافة حقول جديدة لجدول `Users`.

### تعديل جدول `Schools`

سنضيف حقل `parent_school_id` لربط الفروع بالمدرسة الرئيسية. هذا يسمح لنا باستخدام نفس جدول `Schools` لكل من المدارس الرئيسية والفروع، مع تمييز الفروع عن طريق وجود قيمة في هذا الحقل.

| Field Name         | Data Type     | Description                                     | Constraints                                  |
|--------------------|---------------|-------------------------------------------------|----------------------------------------------|
| `parent_school_id` | BIGINT        | معرف المدرسة الرئيسية إذا كان هذا السجل فرعًا. | NULLABLE, FOREIGN KEY (`id` on `schools`)    |
| `status`           | ENUM          | حالة طلب الفرع: `Pending`, `Approved`, `Rejected`, `Modification Requested` | NOT NULL, DEFAULT `Pending`                  |
| `rejection_reason` | TEXT          | سبب الرفض إذا تم رفض طلب الفرع.                 | NULLABLE                                     |

**ملاحظة**: الحقول الأخرى في جدول `Schools` (مثل `name`, `logo_url`, `email`, `address`, `school_type`, `section`, `admin_name`, `website`, `password`, `num_students`, `academic_year`, `country`, `city`, `district`, `contact_number`) ستُستخدم أيضًا لبيانات الفرع.

### تعديل جدول `Users`

سنضيف حقولًا لتمكين إدارة كلمات المرور المؤقتة وتتبع تغييرها.

| Field Name           | Data Type     | Description                                     | Constraints                                  |
|----------------------|---------------|-------------------------------------------------|----------------------------------------------|
| `is_temp_password`   | BOOLEAN       | يشير إلى ما إذا كانت كلمة المرور الحالية مؤقتة. | NOT NULL, DEFAULT `TRUE`                     |
| `last_password_change`| TIMESTAMP     | تاريخ ووقت آخر تغيير لكلمة المرور.              | NULLABLE                                     |
| `branch_id`          | BIGINT        | معرف الفرع الذي ينتمي إليه المستخدم (إذا كان مشرف فرع). | NULLABLE, FOREIGN KEY (`id` on `schools`)    |

**ملاحظة**: الحقل `school_id` الموجود بالفعل في جدول `Users` سيشير إلى `id` المدرسة أو الفرع الذي ينتمي إليه المستخدم.

### جدول `Permissions` (مقترح لتحديد الصلاحيات)

لإدارة الصلاحيات بشكل مرن، خاصةً لمشرف المدرسة الرئيسية للتحكم في صلاحيات الفروع، يمكننا استخدام نظام صلاحيات Laravel (مثل Spatie Laravel Permission) أو إنشاء جدول بسيط.

| Field Name      | Data Type     | Description                                     | Constraints       |
|-----------------|---------------|-------------------------------------------------|-------------------|
| `id`            | BIGINT        | Primary Key, Auto-increment                     | PRIMARY KEY       |
| `user_id`       | BIGINT        | معرف المستخدم (مشرف الفرع).                      | FOREIGN KEY       |
| `permission`    | VARCHAR(255)  | اسم الصلاحية (مثال: `manage_students`, `view_reports`). | NOT NULL          |
| `value`         | BOOLEAN       | قيمة الصلاحية (صحيح/خطأ).                       | NOT NULL          |

## 3. منطق العمل (Business Logic)

### 3.1. إضافة فرع جديد (بواسطة School Admin)

1.  **واجهة المستخدم (UI)**: في صفحة إعدادات مشرف المدرسة، ستكون هناك واجهة لإضافة فرع جديد. ستتضمن هذه الواجهة جميع حقول بيانات المدرسة (الاسم، الشعار، العنوان، إلخ) بالإضافة إلى حقول لإنشاء اسم مستخدم وكلمة مرور مؤقتة لمشرف الفرع الجديد.
2.  **التحقق من الصحة (Validation)**: سيتم التحقق من صحة جميع البيانات المدخلة في الواجهة الأمامية والخلفية.
3.  **إنشاء سجل الفرع**: عند إرسال النموذج، سيتم إنشاء سجل جديد في جدول `Schools` مع تعيين `parent_school_id` إلى `id` المدرسة الرئيسية، وتعيين `status` إلى `Pending`.
4.  **إنشاء حساب مشرف الفرع**: سيتم إنشاء سجل جديد في جدول `Users` لمشرف الفرع الجديد. سيتم تعيين `school_id` إلى `id` الفرع الجديد، و`role` إلى `BranchAdmin`، و`is_temp_password` إلى `TRUE`، وسيتم توليد كلمة مرور مؤقتة وتشفيرها.
5.  **إرسال إشعار**: سيتم إرسال إشعار إلى المشرف العام (Super Admin) بوجود طلب جديد لإضافة فرع.

### 3.2. موافقة المشرف العام (Super Admin Approval)

1.  **واجهة المستخدم (UI)**: في لوحة تحكم المشرف العام، ستظهر قائمة بطلبات الفروع المعلقة.
2.  **عرض التفاصيل**: يمكن للمشرف العام عرض تفاصيل طلب الفرع، بما في ذلك بيانات الفرع وبيانات مشرف الفرع المقترح.
3.  **الإجراءات**: يمكن للمشرف العام اتخاذ أحد الإجراءات التالية:
    *   **الموافقة (Approve)**: يتم تحديث `status` الفرع في جدول `Schools` إلى `Approved`. يتم إرسال بريد إلكتروني إلى مشرف المدرسة الرئيسية ومشرف الفرع الجديد يحتوي على بيانات تسجيل الدخول (اسم المستخدم وكلمة المرور المؤقتة).
    *   **الرفض (Reject)**: يتم تحديث `status` الفرع إلى `Rejected`، ويجب على المشرف العام إدخال `rejection_reason`. يتم إرسال بريد إلكتروني إلى مشرف المدرسة الرئيسية ومشرف الفرع يوضح سبب الرفض.
    *   **طلب تعديل (Request Modification)**: يتم تحديث `status` الفرع إلى `Modification Requested`، ويجب على المشرف العام تحديد التعديلات المطلوبة. يتم إرسال بريد إلكتروني إلى مشرف المدرسة الرئيسية يوضح التعديلات المطلوبة.

### 3.3. تسجيل الدخول الأول وتغيير كلمة المرور (Branch Admin First Login)

1.  **تسجيل الدخول**: عند تسجيل دخول مشرف الفرع لأول مرة باستخدام اسم المستخدم وكلمة المرور المؤقتة، سيتحقق النظام من `is_temp_password`.
2.  **فرض تغيير كلمة المرور**: إذا كانت `is_temp_password` صحيحة، سيتم توجيه المستخدم إلى صفحة إجبارية لتغيير كلمة المرور. يجب عليه إدخال كلمة المرور المؤقتة وكلمة مرور جديدة وتأكيدها.
3.  **تحديث بيانات المستخدم**: بعد تغيير كلمة المرور بنجاح، سيتم تحديث `password` في جدول `Users`، وتعيين `is_temp_password` إلى `FALSE`، وتحديث `last_password_change`.

### 3.4. إدارة الصلاحيات (بواسطة School Admin)

1.  **واجهة المستخدم (UI)**: في صفحة إعدادات مشرف المدرسة الرئيسية، ستكون هناك واجهة لإدارة الفروع وحسابات مشرفي الفروع.
2.  **تحديد الصلاحيات**: يمكن لمشرف المدرسة الرئيسية عرض قائمة الفروع التابعة له وتحديد الصلاحيات لكل مشرف فرع (باستخدام جدول `Permissions` أو حقل `permissions` في جدول `Users`).

## 4. المكدس التقني (Technical Stack)

-   **الخلفية (Backend)**: Laravel (PHP) للتحكم في منطق العمل، التفاعل مع قاعدة البيانات، وإدارة المستخدمين والصلاحيات.
-   **الواجهة الأمامية (Frontend)**: Laravel Blade Templates، Bootstrap 5، JavaScript (Vanilla/jQuery) لتصميم واجهات المستخدم التفاعلية.
-   **قاعدة البيانات (Database)**: MySQL (أو PostgreSQL) لتخزين البيانات.
-   **البريد الإلكتروني (Email)**: نظام البريد في Laravel لإرسال الإشعارات.

## 5. نقاط مهمة للتنفيذ

-   **التحقق من الصلاحيات**: يجب التأكد من أن مشرف المدرسة الرئيسية فقط هو من يمكنه إضافة فروع وإدارة صلاحياتها.
-   **أمان كلمة المرور**: يجب تشفير كلمات المرور المؤقتة وتخزينها بشكل آمن، والتأكد من أن عملية تغيير كلمة المرور آمنة.
-   **التعريب (Localization)**: يجب أن تكون جميع الرسائل والإشعارات قابلة للتعريب (EN/AR).
-   **المرونة**: يجب تصميم نظام الصلاحيات ليكون مرنًا بما يكفي لاستيعاب الصلاحيات المستقبلية.
